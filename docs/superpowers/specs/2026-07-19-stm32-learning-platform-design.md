# STM32 系统学习平台设计规格

- 日期：2026-07-19
- 状态：已完成对话设计确认，等待书面规格复核
- 面向学习者：没有编程基础、希望通过理解和实操系统掌握嵌入式开发的人
- 主控：STM32F103C8T6
- 学习节奏：每周 6–8 小时，共 24 周，可暂停、补修和恢复

## 1. 背景与设计结论

本项目把课程、实验、考核、知识诊断、学习笔记和开发板检测整合到一个 GitHub 友好的仓库中。网页是离线优先的静态应用，不要求注册账号，也不依赖后台服务器。学习进度保存在浏览器中，并可导出 JSON 备份；学习笔记可导出 Markdown 后通过 VS Code 保存到 GitHub。

原始资料的编号范围为 05–49，其中 06 分为 06-1 和 06-2，共 46 份源课程。平台必须为全部 46 份源课程建立明确映射，同时补上零基础学习者需要的电学、二进制、C 语言、工具链和调试基础。

已确认的主要技术决策如下：

1. 使用 React + TypeScript 构建静态网页，发布到 GitHub Pages。
2. 使用 STM32CubeMX 生成工程，以 HAL 为主线；在关键课程加入 LL/寄存器观察，并提供 SPL 到 HAL 的概念映射。
3. 使用 VS Code、STM32 官方扩展、CMake、Ninja 和 STM32CubeCLT 完成编辑、构建、下载和调试。
4. 使用浏览器本地存储保存进度，使用版本化 JSON 做完整备份与恢复。
5. 使用 Web Serial 通过 CH340 与开发板通信，自动记录能够客观验证的实验结果；物理现象由学习者确认。
6. 使用规则驱动的知识标签与补修引擎，不依赖云端 AI 或账号。

## 3. 项目目标与非目标

### 3.1 必须实现的目标

- 引导零基础学习者理解 GPIO、EXTI、NVIC、TIM、ADC、DMA、USART、I2C、SPI、RTC/BKP、PWR、IWDG/WWDG、内部 FLASH 和芯片 ID。
- 每个核心主题都包含概念、CubeMX 配置原因、最小实验、调试观察、故障注入、复述题、测验和笔记任务。
- 覆盖全部 46 份源课程，不把“看完步骤”当作“掌握知识”。
- 用 24 周学习地图显示当前位置、前置知识、预计时间和完成证据。
- 记录概念题、代码/CubeMX 阅读、硬件排错和讲解笔记四类证据。
- 每四周进行阶段闸门考核，诊断薄弱标签并插入补修任务。
- 支持暂停和恢复；支持进度备份、恢复和版本迁移。
- 支持从网页导出 GitHub 友好的 Markdown 学习笔记。
- 支持 CH340 + Web Serial 自动检测，并保留不支持浏览器时的手动评分路径。
- 提供通向 FreeRTOS、数字逻辑、FPGA/PLA、控制和嵌入式系统设计的延伸建议。

### 3.2 明确不做的内容

- 不建立账号、云数据库、多用户班级或在线排名系统。
- 不在浏览器中自动烧录 STM32 固件；固件仍由 VS Code/ST-LINK 完成下载。
- 不声称网页能自动判断 LED 是否真的亮、电机是否真的转或蜂鸣器是否真的发声。
- 不把 SPL 作为主线重新教学；只提供旧 API 到 HAL/LL/寄存器概念的迁移索引。
- 不在本课程内完整实现 FreeRTOS 或 FPGA/PLA 课程，只建立可靠的先修桥梁和后续路线。
- 不为了未来可能出现的需求建设插件系统、后台服务或复杂权限模型。

## 4. 硬件范围与安全边界

课程按用户提供的硬件组合设计，主要包括：

- STM32F103C8T6 最小系统板、ST-LINK、CH340 USB-TTL。
- 面包板、MB102 电源、杜邦线、LED、电阻、按键和电位器。
- 0.96 英寸 OLED、光敏/热敏/红外传感器、旋转编码器。
- 蜂鸣器、舵机、直流电机、风扇、TB6612 电机驱动。
- MPU6050、W25Q64、数码管等模块。

所有实验说明必须带有与当前实验相关的安全提示：

- STM32 GPIO 按 3.3 V 逻辑设计，不把 5 V 电源直接接入普通 IO。
- 电机和舵机使用合适的独立供电，并与 STM32 共地；不由 GPIO 直接驱动负载。
- LED 必须串联限流电阻。
- 接线或改变跳线前先断电，首次上电前运行接线清单。
- W25Q64 自动写入测试只能使用课程保留的测试扇区，不能覆盖学习者数据区。
- 内部 FLASH 自动测试只能使用链接脚本明确保留的页面。
- 涉及低功耗和看门狗复位时，先解释如何恢复和重新连接调试器。

### 5.2 模块边界

平台拆分为可单独理解和测试的模块：

1. **课程内容层**：只描述课程、知识标签、先修关系、实验和引用，不依赖 React 页面。
2. **学习状态层**：保存进度、成绩、证据、笔记和备份版本，不直接操作页面。
3. **评分诊断层**：根据统一规则计算掌握度、阶段结果和补修任务。
4. **设备连接层**：封装串口权限、连接、协议解析、超时和设备测试，不参与课程排版。
5. **网页界面层**：显示学习路径并调用前述模块，不自行复制业务规则。
6. **STM32 固件层**：包含课程工程与统一检测固件，协议版本与网页保持兼容。
7. **内容验证层**：在构建前检查覆盖率、链接、前置关系、评分项和文件结构。

这样可以在不重写课程的情况下更换界面，也可以在不更改评分规则的情况下增加新的实验。

## 6. 内容模型

每一课使用结构化元数据加 Markdown 正文。至少包含：

- 唯一 ID、标题、所属周、预计时间和难度。
- 源课程编号与参考资料链接。
- 先修知识标签、目标知识标签和完成标准。
- “学完后能解释什么”和“学完后能做什么”。
- CubeMX 配置步骤及每一步的原因。
- 接线清单、供电说明、器件保护提示和上电前检查。
- 最小可运行实验、预期现象和可观察寄存器。
- 至少一个故障注入任务，例如错误上下拉、错误波特率或未清中断标志。
- 自动检测项、半自动检测项和人工确认项。
- 概念题、代码/CubeMX 阅读题、排错题和复述提示。
- 笔记模板、复盘问题和跨课程延伸链接。

内容构建器不允许缺失目标、证据或知识标签的课程进入正式发布版本。

## 8. 每周学习闭环

每周遵循同一套可预测流程：

1. **先测**：用 5–10 分钟题目检查先修知识。
2. **理解**：用图、比喻、时序和寄存器解释抽象概念。
3. **配置**：在 CubeMX 中完成最小配置，并解释每个选项为什么存在。
4. **实做**：连接最小硬件，构建、下载并观察现象。
5. **观察**：使用断点、Watch、寄存器、OLED 或串口查看内部状态。
6. **破坏**：主动制造一个典型错误并按证据排查。
7. **复述**：不用照抄资料，以自己的话解释数据流和控制流。
8. **考核**：完成四类评分证据。
9. **记录**：导出 Markdown 笔记并提交到 GitHub。

课程允许暂停。恢复时先进行 10 分钟回忆测验，再决定继续、复习摘要或插入补修。

## 9. 评分、阶段闸门与知识诊断

### 9.1 单课评分

每课满分 100 分：

- 概念理解：25 分。
- CubeMX 配置与代码阅读：25 分。
- 实物实验和故障排查：35 分。
- 复述与学习笔记：15 分。

成绩必须附带证据状态：`自动通过`、`人工确认`、`待验证` 或 `失败`。没有证据的项目不能因为填写了分数而变成“已掌握”。

### 9.2 知识标签掌握度

题目和实验可同时关联多个知识标签。每个标签使用最近最多三份有效证据计算掌握度；较新的证据权重更高，依次为 50%、30%、20%，证据不足时按现有权重归一化。

- 85–100：已掌握。
- 70–84：通过，但需要间隔复习。
- 60–69：需要补修。
- 0–59：需要重新学习核心概念。

### 9.3 每四周阶段闸门

阶段完成需同时满足：

- 本阶段总评平均分不低于 75。
- 所有前置核心标签不低于 70。
- 综合实操成绩不低于 70。

未通过时，后续内容仍可预览，但阶段不能标记为完成。平台从最低的最多三个知识标签中生成补修队列，完成重新测试后再计算闸门。

### 9.4 补修生成规则

- 概念题失分：插入更小的图解、类比、时序拆解和复述题。
- CubeMX/代码题失分：插入“从信号到寄存器再到 HAL”的追踪练习。
- 实操失分：插入最小接线、测量点、串口日志和故障树练习。
- 多个通信协议同时失分：先补同步/异步、主从、帧、时钟和电平等共同基础。
- 连续两次同标签低于 60：回退到其前置标签，不重复堆叠同一道题。

补修是确定性的本地规则，生成原因会显示给学习者，避免不可解释的“黑箱推荐”。

## 10. 跨课程延伸建议

延伸建议由已掌握标签触发：

- 中断 + 定时器 + 状态机掌握后：推荐 FreeRTOS 任务、节拍和并发基础。
- DMA + USART/SPI 掌握后：推荐缓冲区、环形队列、零拷贝和吞吐量分析。
- PWM + ADC + 定时器掌握后：推荐闭环控制、PID 和数字滤波。
- I2C/SPI + 寄存器阅读掌握后：推荐独立阅读数据手册和编写设备驱动。
- GPIO + 时序 + 寄存器掌握后：推荐数字逻辑、有限状态机、Verilog 和 FPGA/PLA。
- PWR + RTC + WDG 掌握后：推荐低功耗系统设计和可靠性工程。

建议只解释下一步为何适合，不把未教授的知识计入当前课程成绩。

## 11. Web Serial 开发板检测

### 11.1 连接方式

默认使用 CH340 的 3.3 V TTL 串口：STM32 USART1 TX（PA9）接 CH340 RX，USART1 RX（PA10）接 CH340 TX，并连接 GND。供电方式只能选择一种，接线页必须提醒检查 CH340 电平和共地。

网页在 Chrome 或 Edge 的安全上下文中运行。学习者点击“连接开发板”后，浏览器才显示串口授权窗口；网页不能绕过用户授权。

### 11.2 模块设计

- `SerialTransport`：端口选择、连接、断开、读写和重连。
- `ProtocolParser`：按行解析 JSON，处理残缺行和非法消息。
- `DeviceTestRunner`：管理测试状态、超时、取消和进度。
- `TestCatalog`：声明每项测试需要的固件、接线和安全条件。
- `EvidenceRecorder`：把结果转换成带来源和时间的课程证据。
- `DeviceSimulator`：在没有开发板时模拟通过、失败、超时、掉线和版本不匹配。

### 11.3 串口协议

协议采用 UTF-8 JSON Lines，即每行一个 JSON 对象。每条命令包含协议版本、请求 ID、消息类型和测试名。

```json
{"v":1,"id":"req-42","type":"run","test":"spi.flash-id","params":{}}
```

设备返回：

```json
{"v":1,"id":"req-42","type":"result","test":"spi.flash-id","status":"pass","value":"EF4017"}
```

所有消息必须限制最大长度。未知版本、未知测试、校验失败和忙状态返回结构化错误，网页显示人话提示而不是原始异常。

### 11.4 可验证范围

| 项目 | 自动或半自动证据 | 必要条件/限制 |
|---|---|---|
| 固件与串口 | 握手、固件版本、芯片 ID、协议版本 | 自动 |
| GPIO | 输出到输入的回环电平变化 | 需要跳线，不能证明 LED 真亮 |
| EXTI | 事件计数和时间戳 | 学习者按键或传感器触发，半自动 |
| TIM/PWM/输入捕获 | PWM 输出接输入捕获，测频率和占空比 | 需要回环跳线 |
| ADC | 值域、变化、稳定性和 DMA 缓冲区 | 电位器由学习者转动；不能替代校准仪器 |
| DMA | 内存搬运结果、计数和完成标志 | 自动 |
| USART | 收发、数据包、错误帧和超时 | 自动 |
| I2C/MPU6050 | WHO_AM_I、寄存器读写、数据变化 | 需要正确模块和接线 |
| SPI/W25Q64 | JEDEC ID、保留扇区写入/读回/恢复 | 只操作明确保留区域 |
| RTC/BKP | 设置、读取、经过时间和复位保持 | 掉电保持需要电池条件 |
| WDG | 复位原因和超时区间 | 半自动，先提示保存状态 |
| 内部 FLASH | 保留页写入/读回和芯片 ID | 只操作链接脚本保留页 |
| PWR | 进入模式、唤醒原因和重连 | 半自动，电流大小需仪表人工确认 |
| LED/蜂鸣器/舵机/电机 | 固件只能报告已执行控制命令 | 亮度、声音、角度和转动必须人工确认 |

### 11.5 失败处理

- 浏览器不支持 Web Serial：解释原因并进入手动证据模式。
- 用户拒绝授权：不反复弹窗，保留“重新连接”按钮。
- 端口掉线：停止当前测试、保留日志，并提示重新插拔和重连。
- 响应超时：显示正在测试的外设、接线和固件要求。
- 协议或固件版本不匹配：禁止运行可能危险的写入测试并给出正确固件路径。
- 页面刷新：串口连接不假定仍存在，进度证据仍保存在本地。

## 12. 网页体验

网页包含以下主要页面：

1. **首页仪表板**：本周目标、下一步、最近成绩、待补修和备份提醒。
2. **24 周学习地图**：显示阶段、前置关系、完成度和闸门状态。
3. **课程页**：概念、配置原因、图示、检查题和扩展阅读。
4. **实验页**：接线清单、构建/下载、观察点、故障注入和完成证据。
5. **开发板检测台**：连接状态、固件版本、测试步骤、日志和结果。
6. **考核页**：概念、代码阅读、实操和复述四类考核。
7. **知识报告**：掌握标签、证据、薄弱原因和补修队列。
8. **笔记与备份**：编辑笔记、导出 Markdown、导出/导入 JSON。

界面默认只突出当前要做的一步。专业术语第一次出现时提供一句人话解释，详细原理放在可展开区域。移动端可用于阅读、测验和进度查看；连接开发板和复杂实验建议使用电脑。

## 13. 本地数据、备份和 GitHub 笔记

### 13.1 数据存储

- IndexedDB 保存课程进度、成绩、证据、补修队列和笔记。
- `localStorage` 只保存主题、当前页面等小型偏好。
- 所有记录包含数据模式版本、内容版本和更新时间。
- 更新应用时先运行迁移；迁移失败时不覆盖原数据。

### 13.2 JSON 备份

导出文件包含进度、成绩、证据、笔记、补修状态和版本信息。导入顺序为：验证结构与版本 → 创建当前数据快照 → 导入到临时区域 → 完整校验 → 原子替换。无效文件不能破坏现有数据。

### 13.3 Markdown 学习笔记

笔记导出格式包含 YAML 元数据、学习目标、接线、CubeMX 决策、代码解释、故障记录、测试证据和复盘。建议保存为：

```text
notes/learner/2026-07-19-w04-gpio-output.md
```

网页不索取 GitHub Token。学习者在 VS Code 的“源代码管理”页面中查看更改、填写说明并提交；课程提供一步一图的简化指引。

## 15. 验证策略

### 15.1 自动验证

- 内容验证器确认 24 周全部存在、46 份源课程均被映射、所有命名外设均有概念/实验/考核。
- 内容验证器检查重复 ID、断链、循环先修关系、缺失知识标签、缺失证据和无效分值。
- 单元测试覆盖评分、标签权重、阶段闸门、补修生成、暂停恢复、数据迁移、导入回滚和串口协议。
- 组件测试覆盖导航、课程完成、补修插入、笔记导出和错误提示。
- 端到端测试覆盖新学习者、跳过已掌握内容、阶段失败补修、备份恢复和手动证据路径。
- 设备模拟器覆盖通过、失败、非法 JSON、分包、超时、掉线、错误固件和协议版本不匹配。
- 所有纳入交付的 STM32 工程必须由 CMake/Ninja 成功构建。
- 网页必须成功构建并在 GitHub Pages 子路径下正确加载资源和路由。

### 15.2 实机验证

发布“实机通过”结论前，至少完成一次 STM32F103C8T6 + CH340 的真实冒烟测试：

1. 烧录检测固件。
2. Chrome/Edge 授权串口并握手。
3. 验证芯片 ID、USART 数据包和断线重连。
4. 使用跳线完成 GPIO 回环和 PWM→输入捕获回环。
5. 连接 MPU6050 读取 WHO_AM_I。
6. 连接 W25Q64 读取 JEDEC ID，并在保留扇区完成写读恢复。
7. 执行一次 RTC/BKP、WDG 复位原因和内部 FLASH 保留页测试。

若当前执行环境无法接触实物，这些项目必须显示为“待实机验证”，模拟器测试不能替代它们。

### 15.3 人工可用性检查

- 零基础用户能从首页找到下一步，并理解首次接线和下载流程。
- 每页在常见电脑宽度和手机宽度下无横向内容丢失。
- 键盘可以完成主要导航和测验，颜色不是唯一状态提示。
- 串口权限、掉线、错误固件和备份失败都有可执行的恢复说明。
- 用导出的 JSON 在全新浏览器资料中恢复后，进度、成绩、证据和笔记一致。
- 用导出的 Markdown 在 GitHub 中查看时结构清晰、链接有效。

## 16. 完成交付标准

只有同时满足以下条件，项目才可宣告完成：

1. 两个指定 skill 已安装、使用并留下验证证据。
2. 24 周课程、46 份源课程映射和全部命名外设覆盖通过内容验证。
3. 每个核心主题至少有一份讲解、一项实验、一项考核和一项故障排查任务。

7. GitHub Pages 部署可打开且关键流程可用。

## 17. 实施分期
