# 第 2 周：C 语言与内存基础

本周的代码先在电脑运行。这样可以把“C 表达式是否正确”与“开发板是否接好”分开，错误更容易观察。目标不是背语法大全，而是建立 STM32 寄存器代码所需的最小模型。

## 学完后能解释

- `uint8_t`、`uint16_t`、`uint32_t` 明确位宽且无符号；`int8_t` 等为有符号整数。位宽决定能保存多少种 bit 组合，符号规则决定这些组合如何解释。
- `&`、`|`、`^`、`~` 和 `<<`/`>>` 是位运算；`&&`、`||` 和 `!` 是逻辑运算，两组不能混用。
- 掩码是只把目标 bit 设为 1 的图样。`x |= mask` 设置位，`x &= ~mask` 清除位，`x & mask` 测试位。
- 函数把输入、输出和副作用划出边界；数组保存同类型连续元素；结构体把不同字段组成一条记录；枚举（`enum`）给有限状态命名。
- 变量的**值**是存储单元里的内容，`&变量` 得到**地址**，指针变量保存一个地址，`*指针` 才访问该地址指向的值。

## 学完后能做到

1. 用固定宽度无符号整数写 set、clear、toggle、test bit 函数。
2. 在纸上逐位追踪一次表达式，不靠运行结果猜答案。
3. 写数组循环、带参数函数、结构体与 enum 状态示例。
4. 打印结构体对象、成员值和地址，区分 `object.member`、`&object.member` 与指针。
5. 开启编译器警告，用测试抓住本应写 `==` 却误写成 `=` 的赋值错误，并显式检查是否移位超出 bit0–bit7 的课程设计范围。

## 概念模型

内存可先想成很多有编号的小格子，地址就是格子编号。类型告诉编译器从某个地址起读多少字节、怎样解释这些 bit。类型不是盒子上的装饰：同样的 `1111_1111`，作为 `uint8_t` 是 255，作为常见二补码 `int8_t` 是 -1。

位操作是“不影响其他开关，只操作指定开关”的工具。假设 `mask = 1u << 3`，得到 `0000_1000`：

- `x | mask`：目标位只要一方为 1 就为 1，因此设置 bit3。
- `x & ~mask`：反掩码只有 bit3 为 0，因此清除 bit3，其他位保持。
- `(x & mask) != 0u`：隔离 bit3 后判断它是否设置。

函数让这条规则只写一次；数组使多个同类型数据能按下标处理；结构体把 `pin`、`mode`、`state` 等字段放在同一对象中；enum 用 `STATE_IDLE` 代替神秘数字。指针则允许函数拿到另一个对象的地址并修改它，力量大，也意味着必须保证地址有效。

## CubeMX 为什么这样配

本周仍不生成 STM32 工程。CubeMX 会产生大量结构体、宏、函数调用和指针参数；如果现在直接照抄，容易把 `GPIO_InitTypeDef` 当成魔法。

先在桌面 C 中看清三个对应关系：CubeMX 的配置最终变成结构体字段；`GPIO_PIN_3` 一类宏通常是位掩码；HAL 函数接收端口指针和参数结构体地址。第 3 周再打开生成代码时，就能从 C 语言含义解释它，而不是只记点击顺序。

## 最小实验

创建一个桌面 `bitmask.c`，启用 `-Wall -Wextra -Wconversion` 等警告。实现：

```c
uint32_t set_bits(uint32_t value, uint32_t mask);
uint32_t clear_bits(uint32_t value, uint32_t mask);
bool test_bits(uint32_t value, uint32_t mask);
```

为每个函数写至少两个 `assert`。再定义：

```c
typedef enum { MODE_INPUT, MODE_OUTPUT } Mode;
typedef struct { uint16_t pin; Mode mode; bool active; } PinConfig;
```

创建一个数组保存两个 `PinConfig`，用函数遍历并打印字段。分别打印 `config.pin`、`&config`、`&config.pin`，记录“值”和“地址”在输出格式与含义上的区别。不要比较不同机器的绝对地址；只解释同一次运行中的关系。

## 调试与寄存器观察

这周没有 MCU 寄存器，但要用调试器或打印完成一次逐步观察：

1. 选择 `value = 0b0010_0101`、`mask = 1u << 3`。
2. 在 OR 前停下，写出两行 bit；单步后确认 bit3 变为 1。
3. 在 AND 反掩码前停下，确认只有 bit3 对应位置为 0。
4. 观察结构体对象地址与成员地址，确认成员位于对象存储范围内，但不要据此假设所有编译器布局完全相同。

以后读取寄存器时，“地址指向外设寄存器、掩码选中字段、位运算改变字段”就是同一模型。

## 故障注入

先复制文件，保证正确版本仍能运行。

- 故障 A：把受控条件 `if (value == expected)` 改为 `if (value = expected)`。`=` 是赋值，表达式结果可能被当成真；`==` 才是比较。查看编译器警告，再添加一个测试让错误必然暴露。
- 故障 B：原本只允许 bit0–bit7，却写 `1u << 12`。该表达式对 32-bit 无符号数本身可能合法，但已经越过**设计允许范围**；如果移位数达到或超过类型宽度，还会进入未定义行为风险。增加范围检查，不要只把 12 改回 7。

修复后重新运行所有断言，说明是哪条假设被破坏。
