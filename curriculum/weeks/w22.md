# 第 22 周：低功耗、看门狗与内部 Flash 可靠性

本周处理“出错后还能否解释和恢复”。安全规则：改接唤醒脚或电流表前必须断电；内部 Flash 只允许最后一页 `0x0800FC00`，禁止 mass erase（全片擦除）和 option bytes（选项字节）操作。功耗、Standby 唤醒和写入中断电均待实机验证。

## 学完后能解释

- Sleep 保留系统时钟配置；Stop 停掉主时钟，唤醒后必须再次调用 `SystemClock_Config`；Standby 类似一次新启动，要检查 `PWR_FLAG_SB` 与 `PWR_FLAG_WU`。
- IWDG 使用独立低速时钟，WWDG 有合法刷新窗口；“循环定时喂狗”不能证明业务健康。
- STM32F103 内部 Flash 先整页 erase，再按 halfword（半字）program，程序必须处理对齐、timeout、错误、readback 和重新 lock。

## 学完后能做到

- 在 `__HAL_RCC_CLEAR_RESET_FLAGS` 之前调用 `CaptureResetFlags` 保存 RCC reset flags。
- 只有显式一次性请求才进入 Sleep/Stop/Standby、启动 watchdog demo 或更新 Flash；上电不会自动 Standby，也不会反复擦写。
- 写新页前保存旧页；新写或 readback 失败时 erase 后 Restore；首次失败但恢复成功返回 `ERROR`，恢复/恢复回读再失败返回 `RESTORE_FAILED`。

## 概念模型

可靠性不是“永不失败”，而是“失败可定位、危险动作有门、旧数据尽量可恢复”。请求门把危险动作从普通 `while(1)` 分开；进展 token 让 watchdog refresh 成为一次性许可；Flash transaction 把 `old -> erase -> new -> readback` 与失败后的 `erase -> old -> readback` 明确分层。

## CubeMX 为什么这样配

PWR/RCC 由 HAL 管理，Stop 返回后恢复 HSE+PLL；IWDG/WWDG 只在显式 demo 中初始化。链接脚本把 FLASH 长度从 64K 缩为 63K，因此最后 1KB 页 `0x0800FC00` 不会被链接器放入代码。擦除类型固定 `FLASH_TYPEERASE_PAGES`、页数 1、program 类型 `FLASH_TYPEPROGRAM_HALFWORD`；绝不设置 `MASSERASE` 或 `OB_*`。

## 最小实验

1. 主机测试先验证一次进展只能换一次喂狗许可。
2. 成功更新偏移 4 的四字节，确认其余半字保持旧值且最终 lock。
3. 注入第 4 次 program 失败，确认第二次 erase 后旧页恢复且返回 `ERROR`。
4. 再让 restore program 失败，确认 `RESTORE_FAILED` 仍最终 lock。
5. 板上低功耗、功耗测量与切电均待实机验证。

## 调试与寄存器观察

上电最先快照 `RCC_CSR`，之后才清 reset flags；记录 PIN/POR/IWDG/WWDG 原因。Standby 返回后读 `PWR_CSR` 的 SB/WU。Flash 观察 `FLASH_SR` 错误位、目标地址、page error 与 readback。Stop 唤醒后比较 `SystemCoreClock`，确认 `SystemClock_Config` 确实执行。

## 故障注入

- 删除 progress report，刷新许可必须为 0；重复调用 take 也必须为 0。
- 让首次 program 失败但 restore 成功，结果不能误报 `RESTORE_FAILED`。
- 让 restore 或 restore readback 失败，结果必须是 `RESTORE_FAILED`，且任何已解锁路径最终 `HAL_FLASH_Lock`。
- 物理掉电不能在未知地址尝试，禁止 mass erase 与 option bytes。

## 复述检查

1. 为什么 Stop 后要恢复 system clock 而 Sleep 不需要？
2. 为什么要先读 reset flags 再清？
3. 为什么 watchdog 只看时间不看进展会掩盖死循环？
