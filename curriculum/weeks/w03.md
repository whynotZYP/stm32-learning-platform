# 第 3 周：STM32 工具链与第一个工程

本周把第 1、2 周的回路、bit、类型和地址接到真实 MCU。重点不是“按一下 Build”，而是能说清每个文件和工具怎样把 C 代码变成 Cortex-M3 可以执行的指令，并把构建证据与实机证据分开。

## 学完后能解释

- Cortex-M3 是 ARM 定义的处理器内核；STM32F103C8T6 是 ST 把这个内核、Flash、SRAM、时钟、GPIO 等外设集成后的具体芯片。
- 存储器映射把 Flash、SRAM 和外设寄存器放进统一地址空间。CPU 访问 GPIO 地址时，读写到的是外设电路，不是普通 RAM。
- 时钟树把振荡源送入 PLL、总线分频器和外设。本工程在确认板上有 8 MHz HSE 晶振后，用 HSE × 9 得到 72 MHz SYSCLK；APB1 再除以 2，保持 36 MHz 上限。
- CubeMX 负责描述芯片、时钟和引脚并生成初始化代码；CMake 描述构建关系；编译器把每个 `.c` 变成 object；链接器结合启动文件和链接脚本得到 ELF；ST-LINK 把映像写入 Flash 并控制调试。
- 生成代码边界的意义：`Core` 中的 HAL 初始化由 CubeMX 维护，学习者逻辑放在 `App/`，必须进入生成文件时只用 `USER CODE` 区域。

## 学完后能做到

1. 核对芯片丝印为 STM32F103C8T6，并核对最小板是否真的装有 8 MHz HSE 晶振和 PC13 LED。
2. 在 CubeMX 选择 STM32F103C8Tx、HSE Crystal/Ceramic Resonator、72 MHz、Serial Wire、PC13 GPIO Output、HAL 与 CMake。
3. 在 VS Code 终端用 CMake preset 配置并由 Ninja 构建，找到生成的 `.elf`。
4. 断电连接 ST-LINK，烧录后在 `main` 停住，查看调用栈、RCC 和 GPIO 寄存器。
5. 根据错误输出诊断一次错误调试器或错误目标，而不是反复插拔碰运气。

## 概念模型

先把芯片看成三层：

1. **内核层**：Cortex-M3 取指、执行、处理中断，拥有通用寄存器、程序计数器等。
2. **存储与总线层**：Flash 保存程序，SRAM 保存运行数据，AHB/APB 总线把内核与外设连接。
3. **外设层**：RCC 产生和分配时钟，GPIO 把内部 bit 与外部引脚连接。

复位后，CPU 从向量表取得初始栈顶和复位处理函数地址，启动文件先建立 C 运行环境，再进入 `main`。`HAL_Init()` 配置 HAL 基础和 SysTick；`SystemClock_Config()` 把时钟切到 HSE+PLL；`MX_GPIO_Init()` 使能端口时钟并配置 PC13；然后 `App_Run()` 周期改变输出。

构建链条要能从左到右复述：

`.c 源码 → 预处理/编译 → .o object → 链接脚本安排 Flash/RAM → ELF → ST-LINK 烧录 Flash → CPU 从复位向量执行`。

ELF 不只是“可执行文件”，还包含调试符号和段信息。成功生成 ELF 证明语法、编译和链接成立，不能证明晶振已起振、芯片已烧录或 LED 真的发光。

## CubeMX 为什么这样配

- **MCU：STM32F103C8Tx**。必须与芯片丝印和封装相符；选错器件会让内存容量、寄存器或引脚映射错误。
- **RCC HSE：Crystal/Ceramic Resonator**。常见最小系统板带 8 MHz 晶振，但图片不能保证每块板都相同。先核对晶振标记或板级原理图；没有晶振时应改用 HSI，并重新计算可达频率，不能照抄 72 MHz。
- **PLL ×9，SYSCLK 72 MHz**。`8 MHz × 9 = 72 MHz`，这是 STM32F103x8 的最高额定系统频率。APB1 设 `/2` 得 36 MHz，APB2 保持 72 MHz。
- **SYS Debug：Serial Wire**。保留 PA13 SWDIO 和 PA14 SWCLK，使用两根调试信号而不是完整 JTAG。
- **PC13：GPIO_Output，初始 High，Low speed**。常见 PC13 板载 LED 为 active-low，高电平先关灯；GPIO speed 描述输出边沿驱动能力，不是本周的闪烁频率。
- **Project Manager：CMake + HAL**。CMake/Ninja 与 STM32CubeCLT 形成可复现命令行构建；HAL 是课程主线。

工程内提交 `.ioc`、CMakeLists、CMakePresets、`Core`、`Drivers`、`cmake`、启动文件、链接脚本和 `App`，使构建不依赖本机绝对的 STM32Cube Repository 路径。

## 最小实验

### 1. 上电前核对

断电。核对芯片、晶振、PC13 LED 和 SWD 丝印。连接 ST-LINK 的 SWDIO→PA13、SWCLK→PA14、GND→GND；供电只选一种。若板上 LED 无法确认，使用外接方案：PC13 接 LED 阴极，LED 阳极经 1 kΩ 接 3.3 V，这仍是低电平点亮。

### 2. 生成与构建

在 CubeMX 保存 `.ioc` 并生成 CMake 工程。项目使用 STM32CubeF1 HAL，`App/app.c` 只引用 `main.h` 生成的 `LED_GPIO_Port` 与 `LED_Pin`，每 `1000 ms` 调用一次 `HAL_GPIO_TogglePin`。

运行 `cmake --preset Debug`，确认日志识别 `arm-none-eabi-gcc`；再运行 `cmake --build --preset Debug --parallel`。记录 Flash/RAM 占用和 ELF 路径。编译失败时先看第一条错误，不用最后一行概括替代根因。

### 3. 烧录与调试

在 VS Code 的 STM32 调试配置中选择实际 ST-LINK 和正确目标。启动后在 `main` 设置断点；停下时确认 PC 指向本工程、调用栈合理。单步越过 `SystemClock_Config` 和 `MX_GPIO_Init`，再继续运行。

## 调试与寄存器观察

在芯片停住时观察：

- RCC `CR`：HSEON/HSERDY、PLLON/PLLRDY 是否与时钟初始化阶段一致。
- RCC `CFGR`：PLL 输入、倍频和系统时钟状态位。
- GPIOC `CRH`：PC13 的 MODE/CNF 字段是否为低速推挽输出。
- GPIOC `ODR`：bit13 是否随程序翻转。

HAL 写引脚会操作寄存器，但调试器显示的是某一暂停时刻。若程序全速运行，寄存器视图刷新可能错过变化；可在 `HAL_GPIO_TogglePin` 后设断点，或降低运行速度。寄存器变化是设备证据，LED 发光仍需人工观察。

## 故障注入

先保存当前能构建的版本，不改变接线。

1. 在调试配置中故意选择一个不存在或错误的探针/目标一次。
2. 保存完整错误信息，区分“未找到探针”“无法读取目标电压”“无法连接内核”“器件描述错误”。
3. 回到证据顺序：确认 ST-LINK 被系统识别 → 断电复核 SWDIO/SWCLK/GND → 上电确认目标电压 → 核对 target → 降低 SWD 速度或 connect under reset（仅在需要时）。
4. 修复后重新停在 `main` 并读取芯片标识，不把“错误提示消失”当作唯一成功标准。

不要通过随机交换线寻找答案，PA13 与 PA14 必须按信号名连接。
